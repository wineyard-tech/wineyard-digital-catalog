# Part 2: Place Order + Zoho Sales Order

## 1. CONTEXT

Implement "Place Order" feature for registered users. When user confirms their cart in the app, create a Sales Order in Zoho Books (converting from Estimate if exists), store it in Supabase, and send WhatsApp order confirmation.

**User Type:** Registered users only (have `zoho_contact_id` in auth metadata)
**Trigger:** "Place Order" button in cart/checkout page
**Outcome:** Zoho Sales Order created → Supabase record created → WhatsApp confirmation sent

**Two Scenarios:**
1. User came from estimate (via deep link) → Convert estimate to sales order
2. User placing order directly → Create fresh sales order

**Tech Stack:**
- Zoho Books API (sales order creation, estimate conversion)
- Supabase (sales order storage)
- WhatsApp Business API (WABA template: `wineyard_salesorder`)

## 3. TASKS

### 3.1 Database Schema

Create Supabase table: `sales_orders`

**Columns:**
- `id` (uuid, primary key)
- `zoho_salesorder_id` (text, unique, nullable) - from Zoho API, NULL until Zoho sync succeeds
- `zoho_salesorder_number` (text, nullable) - format: SO-XXXXXX, NULL until Zoho sync succeeds
- `zoho_estimate_id` (text, nullable) - if converted from estimate
- `estimate_id` (uuid, nullable) - references estimates.id (Supabase estimate record)
- `zoho_contact_id` (text, not null)
- `user_id` (uuid, not null) - references auth.users
- `line_items` (jsonb, not null) - order items snapshot
- `cart_hash` (text, not null) - hash of line_items for duplicate detection
- `subtotal` (numeric, not null)
- `total` (numeric, not null)
- `status` (text, not null) - 'pending_zoho_sync', 'confirmed', 'invoiced', 'delivered', 'cancelled', 'failed'
- `zoho_sync_attempts` (integer, default 0) - retry counter
- `zoho_sync_error` (text, nullable) - last Zoho API error message
- `whatsapp_sent` (boolean, default false)
- `whatsapp_message_id` (text, nullable)
- `created_at` (timestamptz, default now())
- `updated_at` (timestamptz, default now())

**Indexes:**
- `zoho_salesorder_id` (unique, where not null)
- `user_id` (for user's order history)
- `zoho_estimate_id` (for estimate-to-order tracking)
- `estimate_id` (for Supabase estimate reference)
- `zoho_contact_id` (for contact lookup)
- `status` (for background job queries)
- `cart_hash` (for duplicate detection)

**Update `estimates` table:**
Add columns:
- `converted_to_salesorder_id` (uuid, nullable) - references sales_orders.id (Supabase SO record)
- `zoho_converted_to_salesorder_id` (text, nullable) - Zoho SO ID (may be NULL if conversion pending)

**RLS:** Authenticated users can read their own sales orders only.

### 3.3 Zoho Sales Order Service

Create `/lib/zoho/salesorders.ts`

**Function 1: `createSalesOrder(contactId, lineItems, estimateId?, userMetadata)`**

**Scenario A: Fresh Sales Order (no estimate)**

**Zoho API Call:**
```
POST https://www.zohoapis.in/books/v3/salesorders
Headers: Authorization: Zoho-oauthtoken {token}
Query: organization_id={org_id}

Body:
{
  customer_id: contactId,
  salesorder_number: (auto-generated by Zoho),
  reference_number: (optional),
  date: YYYY-MM-DD (today),
  shipment_date: (optional, can be null),
  line_items: [...],
  notes: "Order placed via WineYard Digital Catalog",
  status: "confirmed"
}
```

**Scenario B: Convert Estimate to Sales Order**

**Zoho API Call:**
```
POST https://www.zohoapis.in/books/v3/estimates/{estimate_id}/convertToSalesOrder
Headers: Authorization: Zoho-oauthtoken {token}
Query: organization_id={org_id}

Body: (usually empty, Zoho converts automatically)
```

**Return (both scenarios):**
```typescript
{
  zoho_salesorder_id: string,
  salesorder_number: string,     // SO-XXXXXX from Zoho
  total: number,
  line_items: array
}
```

**Error Handling:**
- Retry once on network failure
- Handle estimate already converted error (if user clicks multiple times)
- Log all Zoho API errors
- Return structured error for UI display

### 3.5 API Route

Create `/app/api/orders/create/route.ts` (POST)

**Request Body:**
```typescript
{
  cartItems: Array<{
    zoho_item_id: string,
    name: string,
    quantity: number,
    rate: number
  }>,
  estimateId?: string           // Optional: if converting from estimate
}
```

**Logic:**
1. Verify user is authenticated (get session)
2. Extract `zoho_contact_id` from user metadata
3. If missing → return 403 "Registration required"
4. Calculate totals from cart items
5. **Generate cart hash** (hash of sorted line_items JSON)
6. **Check for duplicate:** Query Supabase for sales_order with same `cart_hash` + `user_id` + `created_at > now() - 1 hour` + `status != 'failed'`
   - If found: Return error "Order already placed for this cart"
7. **If `estimateId` provided (converting from estimate):**
   - Fetch estimate from Supabase by `id` (uuid)
   - Verify estimate belongs to user
   - Check if already converted (`converted_to_salesorder_id` is not null)
   - If already converted → return error "Order already placed for this quote"
   - Store `estimate.zoho_estimate_id` for Zoho conversion API
8. **Create Supabase sales_order record FIRST:**
   ```
   INSERT INTO sales_orders:
   - user_id, zoho_contact_id
   - line_items, cart_hash
   - subtotal, total
   - status: 'pending_zoho_sync'
   - estimate_id: {estimateId} (if converting)
   - zoho_estimate_id: {from estimate} (if converting)
   - zoho_salesorder_id: NULL
   - zoho_salesorder_number: NULL
   - zoho_sync_attempts: 0
   ```
   Get the generated `id` (uuid) from Supabase response
9. **Call Zoho API:**
   - **If converting from estimate:** Use Zoho convert API with `estimate.zoho_estimate_id`
     - **Important:** Only call if `estimate.zoho_estimate_id` is NOT NULL (estimate must have synced successfully)
     - If `zoho_estimate_id` is NULL: Fall back to creating fresh SO (estimate sync hasn't completed yet)
   - **If fresh order:** Use Zoho create sales order API
   - **If Zoho API fails:** Retry once after 2 seconds
   - **If still fails after retry:**
     - Update Supabase sales_order:
       - `zoho_sync_attempts = 1`
       - `zoho_sync_error = {error message}`
       - `status = 'pending_zoho_sync'` (unchanged)
     - Return to user: `{success: false, error: "Order saved but sync failed. We'll retry shortly.", orderId: {supabase_id}}`
     - Background job will retry later
10. **If Zoho API succeeds:**
    - Update Supabase sales_order:
      - `zoho_salesorder_id = {from Zoho response}`
      - `zoho_salesorder_number = {SO-XXXXXX from Zoho}`
      - `status = 'confirmed'`
      - `zoho_sync_attempts = 1`
    - If converted from estimate, update estimate record:
      - `converted_to_salesorder_id = {supabase_sales_order_id}`
      - `zoho_converted_to_salesorder_id = {zoho_salesorder_id}`
11. Format line items for WhatsApp template
12. Generate deep link: `{APP_URL}/orders/{supabase_sales_order_id}` (use Supabase ID, not Zoho ID)
13. Send WhatsApp confirmation
14. Update sales_order record: `whatsapp_sent = true`, `whatsapp_message_id = {id}`
15. Return: `{success: true, salesOrderNumber: "SO-XXXXXX", orderId: {supabase_id}}`

**Response:**
- 200: Order created and confirmed (or sync pending with retry scheduled)
- 400: Invalid cart data, estimate already converted, or duplicate order
- 403: User not registered
- 500: Unrecoverable error

### 3.9 Estimate Update API (for cart changes)

Create `/app/api/quotes/update/route.ts` (PUT)

**Request Body:**
```typescript
{
  estimateId: string,           // Supabase estimate ID (uuid)
  cartItems: Array<{...}>       // Updated cart items
}
```

**Logic:**
1. Fetch estimate from Supabase by `id`
2. Verify estimate belongs to user
3. Verify not yet converted to sales order (`converted_to_salesorder_id` is null)
4. Calculate new totals, generate new cart_hash
5. **If `zoho_estimate_id` exists (estimate synced to Zoho):**
   - Call Zoho API to update estimate line items
   - If Zoho update succeeds: Update Supabase with new line items
   - If Zoho update fails: Keep Supabase as-is, log error
6. **If `zoho_estimate_id` is NULL (estimate not yet synced):**
   - Just update Supabase record with new line items
   - Background job will sync to Zoho later
7. Update Supabase: `line_items`, `cart_hash`, `subtotal`, `total`, `updated_at`
8. Return: `{success: true}`

**Zoho API Call (if estimate synced):**
```
PUT https://www.zohoapis.in/books/v3/estimates/{zoho_estimate_id}
Body: { line_items: [...updated items...] }
```

### 3.10 Background Retry Job (Optional - Can Defer to Phase 2)

Create `/lib/cron/retry-zoho-salesorders.ts` or use Vercel Cron

**Schedule:** Every 5 minutes

**Logic:**
1. Query Supabase for sales_orders where:
   - `status = 'pending_zoho_sync'`
   - `created_at > now() - 24 hours`
   - `zoho_sync_attempts < 10`
2. For each sales order:
   - Increment `zoho_sync_attempts`
   - Determine if converting from estimate or creating fresh:
     - If `zoho_estimate_id` is NOT NULL: Use Zoho convert API
     - If `zoho_estimate_id` is NULL: Use Zoho create SO API
   - **If success:**
     - Update with Zoho IDs (`zoho_salesorder_id`, `zoho_salesorder_number`)
     - Change `status = 'confirmed'`
     - Update linked estimate (if exists) with `converted_to_salesorder_id`
     - Send WhatsApp confirmation (if not already sent)
   - **If fail:**
     - Update `zoho_sync_error` with latest error
     - Keep `status = 'pending_zoho_sync'`
     - Will retry in next cron cycle
3. For sales orders older than 24 hours with `status = 'pending_zoho_sync'`:
   - Change `status = 'failed'`
   - Send alert to admin WhatsApp: "Sales Order sync failed after 24hrs: Order ID {id}, User: {phone}"

**Vercel Cron Setup (if using):**
```
// vercel.json
{
  "crons": [
    {
      "path": "/api/cron/retry-zoho-estimates",
      "schedule": "*/5 * * * *"
    },
    {
      "path": "/api/cron/retry-zoho-salesorders",
      "schedule": "*/5 * * * *"
    }
  ]
}
```

## NOTES

- **CRITICAL: Create Supabase record FIRST, then Zoho.** This enables retry capability.
- Sales order number is auto-generated by Zoho
- Zoho estimate conversion API only works if estimate already exists in Zoho (has `zoho_estimate_id`)
- If estimate hasn't synced to Zoho yet (`zoho_estimate_id` is NULL), create fresh SO instead of converting
- Use Supabase `id` (uuid) for deep links, not Zoho ID (since Zoho ID may be NULL during pending sync)
- WhatsApp only sent after successful Zoho sync (status = 'confirmed')
- If Zoho sync pending, user can still see order in app but WhatsApp not sent yet
- Cart hash prevents duplicate orders for identical carts within 1 hour
- If estimate already converted, Zoho returns error - handle gracefully with user-friendly message
- Background retry job is optional for MVP - can implement in Phase 2
- WhatsApp template must be pre-approved in WABA dashboard
- Test with Zoho sandbox/test organization first
- Stock availability ignored in this phase (manual review at invoice stage)
